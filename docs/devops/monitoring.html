Monitoring & Logging

Effective monitoring and logging are crucial for maintaining reliable systems. This guide covers observability practices, tools, and strategies for comprehensive system monitoring and log management.

Observability Fundamentals

The Three Pillars

Metrics - What Happened?

Quantitative measurements of system performance and behavior over time.

  • CPU, memory, disk usage
  • Response times and throughput
  • Error rates and success rates
  • Business KPIs

Logs - Why Did It Happen?

Detailed records of events, errors, and system activities with timestamps.

  • Application logs
  • System logs
  • Audit logs
  • Security events

Traces - How Did It Happen?

Distributed tracing shows the path of requests through your system.

  • Request flows
  • Service dependencies
  • Performance bottlenecks
  • Debugging complex issues

Application Metrics

Instrument Your Code

# Python Flask app with Prometheus metrics
from flask import Flask
from prometheus_flask_exporter import PrometheusMetrics

app = Flask(__name__)
metrics = PrometheusMetrics(app)

# Static information as metric
metrics.info('app_info', 'Application info', version='1.0.0')

@app.route('/')
@metrics.counter('requests_total', 'Total requests', labels={'endpoint': 'home'})
def home():
    return 'Hello World!'

@app.route('/health')
@metrics.do_not_track()
def health():
    return {'status': 'healthy'}

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)
# Node.js Express app
const express = require('express');
const promBundle = require('express-prom-bundle');

const app = express();
const metricsMiddleware = promBundle({
  includeMethod: true,
  includePath: true,
  includeStatusCode: true,
  includeUp: true,
  customLabels: { project_name: 'myapp' },
  promClient: { collectDefaultMetrics: {} }
});

app.use(metricsMiddleware);

// Routes
app.get('/', (req, res) => {
  res.send('Hello World!');
});

app.get('/health', (req, res) => {
  res.json({ status: 'healthy' });
});

app.listen(8080, () => {
  console.log('Server running on port 8080');
});

Logstash Configuration

Log Processing Pipeline

# logstash.conf
input {
  beats {
    port => 5044
  }

  tcp {
    port => 5000
    codec => json
  }
}

filter {
  if [type] == "apache" {
    grok {
      match => { "message" => "%{COMBINEDAPACHELOG}" }
    }
    date {
      match => [ "timestamp", "dd/MMM/yyyy:HH:mm:ss Z" ]
      target => "@timestamp"
    }
    geoip {
      source => "clientip"
    }
  }

  if [type] == "nginx" {
    grok {
      match => { "message" => "%{NGINXACCESS}" }
    }
    mutate {
      convert => { "bytes" => "integer" }
      convert => { "response" => "integer" }
    }
  }

  if [type] == "app" {
    json {
      source => "message"
    }
    mutate {
      remove_field => ["message"]
    }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "%{[@metadata][beat]}-%{[@metadata][version]}-%{+YYYY.MM.dd}"
  }

  stdout {
    codec => rubydebug
  }
}

Prometheus Alert Rules

Define Alert Conditions

# alert_rules.yml
groups:
- name: example
  rules:

  # Instance down alert
  - alert: InstanceDown
    expr: up == 0
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "Instance {{ $labels.instance }} down"
      description: "{{ $labels.instance }} of job {{ $labels.job }} has been down for more than 5 minutes."

  # High CPU usage
  - alert: HighCpuUsage
    expr: 100 - (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 90
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High CPU usage on {{ $labels.instance }}"
      description: "CPU usage is {{ $value }}%"

  # Low disk space
  - alert: LowDiskSpace
    expr: (node_filesystem_avail_bytes / node_filesystem_size_bytes) < 0.1
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Low disk space on {{ $labels.instance }}"
      description: "Disk space is below 10%"

  # High memory usage
  - alert: HighMemoryUsage
    expr: (1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) > 0.9
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High memory usage on {{ $labels.instance }}"
      description: "Memory usage is {{ $value }}%"

  # Application errors
  - alert: HighErrorRate
    expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.1
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "High error rate on {{ $labels.instance }}"
      description: "Error rate is {{ $value }}"

Best Practices

✅ Monitoring Best Practices:
  • Use the RED method for services: Rate, Errors, Duration
  • Use the USE method for resources: Utilization, Saturation, Errors
  • Implement proper log levels (DEBUG, INFO, WARN, ERROR)
  • Structure logs as JSON for better parsing
  • Set up alerts for symptoms, not causes
  • Use distributed tracing for microservices
  • Implement log rotation to prevent disk space issues
  • Create runbooks for common alerts
  • Monitor business metrics, not just technical metrics
  • Implement proper retention policies for logs and metrics
⚠️ Common Monitoring Mistakes:
  • Alert fatigue from too many alerts
  • Missing context in alerts and logs
  • Not monitoring business KPIs
  • Poor log message quality
  • Storing sensitive data in logs
  • No alert escalation procedures
  • Ignoring monitoring during development
  • Not testing monitoring systems

Implementation Strategy

Start Small and Scale

  1. Week 1: Set up basic infrastructure monitoring (CPU, memory, disk)
  2. Week 2: Implement application health checks and basic metrics
  3. Week 3: Set up centralized logging for critical applications
  4. Week 4: Create basic alerts and notification channels
  5. Month 2: Implement distributed tracing and advanced dashboards
  6. Month 3: Set up business metrics monitoring and SLOs

Define SLOs and SLIs

Service Level Objectives

# Example SLOs
# API Availability: 99.9% uptime
# API Latency: 95% of requests < 500ms
# Error Rate: < 1% of requests return errors
# Data Freshness: < 5 minutes lag

# SLI calculations
# Availability = (total time - downtime) / total time
# Latency = percentile of response times
# Error Rate = (error responses / total responses) * 100
# Freshness = time since last successful update
