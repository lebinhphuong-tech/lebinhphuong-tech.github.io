Ansible Core Concepts

Understanding Ansible core concepts is essential for writing effective automation. This guide covers the fundamental building blocks of Ansible.

Inventory

What is Inventory?

Inventory defines the hosts and groups of hosts that Ansible will manage. It contains connection details, variables, and organizes managed nodes into logical groups.

Inventory File Structure:

# /etc/ansible/hosts or hosts.ini
[webservers]
web1 ansible_host=192.168.1.10 ansible_user=ubuntu
web2 ansible_host=192.168.1.11 ansible_user=ubuntu

[databases]
db1 ansible_host=192.168.1.20 ansible_user=centos

[production:children]
webservers
databases

[all:vars]
ansible_python_interpreter=/usr/bin/python3

Dynamic Inventory:

Ansible supports dynamic inventory from cloud providers:

# AWS EC2 dynamic inventory
plugin: aws_ec2
regions:
  - us-east-1
filters:
  instance-state-name: running
keyed_groups:
  - key: tags.Environment
    prefix: env

Common Commands:

# Test connectivity to all hosts
ansible all -m ping -i inventory.ini

# List all hosts in inventory
ansible all --list-hosts -i inventory.ini

# Show inventory variables
ansible web1 -m debug -a 'var=hostvars[inventory_hostname]' -i inventory.ini

Playbooks

What is a Playbook?

Playbooks are YAML files that define a series of tasks to be executed on managed hosts. They are the primary way to orchestrate complex automation workflows.

Basic Playbook Structure:

---
- name: Install and configure web server
  hosts: webservers
  become: yes
  gather_facts: yes

  vars:
    http_port: 80
    max_clients: 200

  tasks:
    - name: Update package cache
      apt:
        update_cache: yes
      when: ansible_os_family == "Debian"

    - name: Install nginx
      apt:
        name: nginx
        state: present
      when: ansible_os_family == "Debian"

    - name: Start nginx service
      service:
        name: nginx
        state: started
        enabled: yes

  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted
      listen: restart_webserver

Playbook Components:

  • name - Descriptive name for the play
  • hosts - Target hosts to run against
  • become - Run with elevated privileges
  • vars - Variables for the play
  • tasks - List of tasks to execute
  • handlers - Tasks triggered by other tasks

Tasks

What is a Task?

Tasks are the individual units of work in Ansible. Each task executes a module with specific parameters and can include conditionals, loops, and error handling.

Task Examples:

tasks:
  # Simple task
  - name: Install package
    package:
      name: nginx
      state: present

  # Task with conditional
  - name: Configure firewall
    ufw:
      rule: allow
      port: "80"
      proto: tcp
    when: ansible_distribution == "Ubuntu"

  # Task with loop
  - name: Install multiple packages
    package:
      name: "{{ item }}"
      state: present
    loop:
      - nginx
      - curl
      - vim

  # Task with register
  - name: Check service status
    command: systemctl is-active nginx
    register: nginx_status
    ignore_errors: yes

  - name: Debug service status
    debug:
      var: nginx_status.stdout

Task Execution Control:

tasks:
  - name: Critical task
    command: /bin/false
    ignore_errors: no  # Fail on error (default)

  - name: Optional task
    command: systemctl status myservice
    ignore_errors: yes  # Continue on error

  - name: Idempotent task
    file:
      path: /tmp/test
      state: touch
    changed_when: false  # Never report as changed

Modules

What are Modules?

Modules are reusable units of code that Ansible executes on managed nodes. They handle the actual work of configuration management, package installation, and system administration.

Common Module Categories:

  • System - User, group, service management
  • Package - apt, yum, pip, npm
  • Files - copy, template, file, lineinfile
  • Commands - command, shell, script
  • Cloud - AWS, Azure, GCP modules
  • Database - MySQL, PostgreSQL, MongoDB

Module Examples:

# Package management
- apt: name=nginx state=present
- yum: name=httpd state=latest
- pip: name=django state=present

# File operations
- copy: src=config dest=/etc/app/config
- template: src=nginx.j2 dest=/etc/nginx/nginx.conf
- file: path=/tmp/cache state=directory mode=0755

# Service management
- service: name=nginx state=started enabled=yes

# User management
- user: name=deploy state=present groups=wheel
- group: name=developers state=present
💡 Tip: Use ansible-doc module_name to get detailed documentation for any module.

Roles

What is a Role?

Roles are a way to organize playbooks into reusable components. They encapsulate variables, tasks, files, templates, and handlers into a standardized directory structure.

Role Directory Structure:

roles/
├── nginx/
│   ├── tasks/
│   │   └── main.yml
│   ├── handlers/
│   │   └── main.yml
│   ├── vars/
│   │   └── main.yml
│   ├── defaults/
│   │   └── main.yml
│   ├── files/
│   ├── templates/
│   ├── meta/
│   │   └── main.yml
│   └── README.md

Role Example:

# roles/nginx/tasks/main.yml
---
- name: Install nginx
  package:
    name: nginx
    state: present

- name: Configure nginx
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: restart nginx

- name: Enable nginx service
  service:
    name: nginx
    state: started
    enabled: yes

# roles/nginx/handlers/main.yml
---
- name: restart nginx
  service:
    name: nginx
    state: restarted

Using Roles in Playbook:

---
- name: Configure web server
  hosts: webservers
  roles:
    - nginx
    - { role: mysql, db_name: myapp }
    - { role: monitoring, when: env == "production" }

Variables

What are Variables?

Variables store values that can be used throughout playbooks. They allow for flexibility and reusability in Ansible automation.

Variable Sources:

  • Playbook vars - Defined in playbook
  • Role vars - Defined in roles
  • Inventory vars - Host/group variables
  • Extra vars - Command line variables
  • Facts - Gathered from managed hosts

Variable Examples:

# Playbook variables
vars:
  app_name: myapp
  app_port: 8080
  db_host: "{{ groups['databases'][0] }}"

# Group variables (group_vars/webservers.yml)
---
nginx_version: "1.20"
php_version: "8.1"

# Host variables (host_vars/web1.yml)
---
ansible_host: 192.168.1.10
server_role: frontend

# Using variables
- name: Install specific version
  apt:
    name: "nginx={{ nginx_version }}"
    state: present

- name: Configure application
  template:
    src: app.conf.j2
    dest: /etc/app/config.yml
  vars:
    port: "{{ app_port }}"
    database_url: "{{ db_host }}"

Variable Precedence:

  1. Extra vars (highest priority)
  2. Task vars
  3. Block vars
  4. Role vars
  5. Play vars
  6. Host vars
  7. Group vars
  8. Defaults (lowest priority)

Templates

What are Templates?

Templates use Jinja2 templating engine to generate configuration files dynamically. They combine static content with variables to create customized configurations.

Jinja2 Template Example:

# templates/nginx.conf.j2
server {
    listen {{ nginx_port | default(80) }};
    server_name {{ server_name | default('localhost') }};

    {% if ssl_enabled | default(false) %}
    listen 443 ssl;
    ssl_certificate {{ ssl_cert_path }};
    ssl_certificate_key {{ ssl_key_path }};
    {% endif %}

    location / {
        proxy_pass http://{{ app_host }}:{{ app_port }};
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    {% for location in additional_locations | default([]) %}
    location {{ location.path }} {
        {{ location.config | indent(8) }}
    }
    {% endfor %}
}

Using Templates in Tasks:

tasks:
  - name: Configure nginx
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/sites-available/app
      mode: '0644'
      backup: yes
    vars:
      nginx_port: 80
      server_name: "{{ inventory_hostname }}"
      app_host: "127.0.0.1"
      app_port: 8080
    notify: reload nginx

Jinja2 Filters:

# String filters
{{ variable | upper }}           # Convert to uppercase
{{ variable | lower }}           # Convert to lowercase
{{ variable | title }}           # Title case
{{ variable | replace('old','new') }}  # Replace text

# List filters
{{ mylist | join(',') }}         # Join list items
{{ mylist | first }}             # First item
{{ mylist | last }}              # Last item
{{ mylist | length }}            # List length

# Default values
{{ variable | default('default_value') }}
{{ variable | default(80, true) }}  # Treat empty string as false

Facts

What are Facts?

Facts are variables that contain information about the managed hosts. Ansible automatically gathers facts about system properties, network interfaces, and installed packages.

Common Facts:

# System facts
ansible_hostname                    # Hostname
ansible_os_family                   # OS family (RedHat, Debian)
ansible_distribution               # Distribution (Ubuntu, CentOS)
ansible_architecture               # CPU architecture

# Network facts
ansible_default_ipv4.address        # Default IPv4 address
ansible_all_ipv4_addresses          # All IPv4 addresses
ansible_interfaces                  # Network interfaces

# Hardware facts
ansible_processor_count             # CPU count
ansible_memtotal_mb                 # Total memory in MB
ansible_mounts                      # Mounted filesystems

Using Facts in Tasks:

tasks:
  - name: Install on Ubuntu
    apt:
      name: apache2
      state: present
    when: ansible_os_family == "Debian"

  - name: Install on CentOS
    yum:
      name: httpd
      state: present
    when: ansible_os_family == "RedHat"

  - name: Configure based on memory
    template:
      src: myapp.conf.j2
      dest: /etc/myapp/config
    vars:
      worker_processes: "{{ ansible_processor_count }}"
      max_memory: "{{ ansible_memtotal_mb }}"

Custom Facts:

# Create custom fact file
- name: Create custom fact
  copy:
    content: |
      [application]
      version=1.2.3
      environment=production
    dest: /etc/ansible/facts.d/app.fact
    mode: '0644'

# Use custom fact
- name: Debug custom fact
  debug:
    msg: "App version: {{ ansible_local.app.application.version }}"

Conditionals and Loops

Conditional Execution

Conditionals allow tasks to run only when certain conditions are met. They use Jinja2 expressions and can reference variables, facts, and registered results.

Conditional Examples:

tasks:
  - name: Install nginx on Ubuntu
    apt:
      name: nginx
      state: present
    when: ansible_distribution == "Ubuntu"

  - name: Configure firewall
    ufw:
      rule: allow
      port: "{{ item }}"
      proto: tcp
    loop:
      - 80
      - 443
    when: ansible_os_family == "Debian"

  - name: Upgrade packages
    apt:
      upgrade: dist
    when:
      - ansible_distribution == "Ubuntu"
      - upgrade_packages | default(false)

  - name: Check service status
    command: systemctl status nginx
    register: nginx_status
    ignore_errors: yes

  - name: Start nginx if not running
    service:
      name: nginx
      state: started
    when: nginx_status.rc != 0

Loop Constructs:

# Simple loop
- name: Install packages
  package:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - curl
    - vim

# Loop with dictionary
- name: Create users
  user:
    name: "{{ item.name }}"
    state: present
    groups: "{{ item.groups }}"
  loop:
    - { name: 'alice', groups: 'wheel' }
    - { name: 'bob', groups: 'developers' }

# Loop with conditional
- name: Configure services
  service:
    name: "{{ item.name }}"
    state: "{{ item.state }}"
  loop:
    - { name: 'nginx', state: 'started' }
    - { name: 'mysql', state: 'stopped' }
  when: item.state == 'started'
